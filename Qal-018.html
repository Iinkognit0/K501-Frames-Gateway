This chapter consolidates the formal definition of QAL into a compact, implementation-ready reference. It is normative.

18.1 Scope of the Grammar


QAL specifies:

  • canonical structure,

  • serialization order,

  • hash scope boundaries.


QAL does not specify semantics, execution, or interpretation.

18.2 Lexical Conventions

  • Encoding: UTF-8 (no BOM)

  • Line endings: LF (\n)

  • Whitespace: significant only where explicitly allowed

  • Numerals: base-10 unless stated otherwise

  • Hex: lowercase, fixed-length

18.4 High-Level Structure


A QAL artifact is a Frame.

Frame ::= Header Payload Hash

Order is mandatory and immutable.

18.5 Header Grammar (EBNF)

Header ::= "QAL" Version HeaderFields
Version ::= uint64

HeaderFields ::= { Field }
Field ::= FieldName ":" FieldValue "\n"

FieldName ::= string
FieldValue ::= string | uint64 | bool

Rules:

  • Field order is canonical.

  • No duplicate field names.

  • Header terminates before payload begins.

18.6 Payload Grammar

Payload ::= PayloadLength PayloadBytes
PayloadLength ::= uint64
PayloadBytes ::= bytes

Rules:

  • Payload is opaque.

  • No internal structure is assumed by QAL.

18.7 Hash Grammar

Hash ::= HashAlg ":" HashValue "\n"
HashAlg ::= "sha256" | "sha3-256"
HashValue ::= hex256

Rules:

  • Hash covers Header + Payload, byte-exact.

  • Hash algorithm must be explicitly declared.

18.8 Canonical Serialization Order

  1. Magic + version

  2. Header fields (canonical order)

  3. Payload length

  4. Payload bytes

  5. Hash line


Any deviation invalidates the artifact.

18.10 Validation Checklist


An implementation MUST verify:

  1. Grammar conformance

  2. Canonical ordering

  3. Payload length correctness

  4. Hash recomputation match

  5. Append-only placement (contextual)

18.12 Minimal Compliance Statement


A system is QAL-compliant iff it can:

  • parse this grammar,

  • reproduce hashes,

  • emit byte-identical artifacts.


Nothing else is required.

18.13 Stability Note


This grammar is freeze-ready.

Future versions may extend via new versions only, never mutation.
